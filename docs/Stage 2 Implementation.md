# FounderLens — Stage 2 Interview: Implementation Plan

> This document is the complete reference for building Stage 2 of FounderLens.
> It covers the mental model, architecture, data flow, prompts, and build order.
> The coding agent should use this as a thinking guide — not a rigid spec.
> Adapt where code reality demands it.

---

## 1. What Stage 2 Is

Stage 2 is a **stateful, cluster-driven interview loop** built on Streamlit's rerun model.

The founder answers one question at a time. Questions are grouped into thematic **clusters** — each cluster covers one of 7 dimensions of the startup idea. The AI generates each cluster upfront as a batch, the founder works through it question by question, and when the last answer is submitted the whole cluster is evaluated together in one LLM call.

The session ends when all 7 dimensions are marked covered. At that point the stage transitions to `"synthesis"`.

The entire interview runs through two files:
- `interview_engine.py` — all logic, all LLM calls, no Streamlit
- `ui/interview.py` — all rendering, reads from session_state, no business logic

---

## 2. The 7 Dimensions

These are the topics the interview must cover before it can end. They map to canvas field groups.

| Dimension Key | Topic | Primary Canvas Fields Targeted |
|---|---|---|
| `problem_discovery` | Is the problem real, painful, frequent? | `idea.problem`, `audience.pain_description` |
| `target_audience` | Who exactly is the early adopter? | `audience.primary_segment`, `audience.early_adopter_profile` |
| `market_timing` | Is the market big enough? Why now? | `market.timing_signal`, `market.size_bucket` |
| `solution_pressure` | Is the solution meaningfully better? | `solution.description`, `solution.key_differentiator` |
| `competitive_reality` | What do people use today? What's the edge? | `competition.alternatives`, `competition.unfair_advantage` |
| `business_model` | How does money flow? | `business_model.revenue_type`, `business_model.who_pays` |
| `founder_market_fit_risk` | Why this founder? What's the riskiest assumption? | `founder.background_relevance`, `risks[].assumption` |

## 3. Session State Shape

Stage 2 uses two layers of state:

**Layer 1 — FounderCanvas** (`st.session_state.canvas`)
This is the persistent data model shared across all stages. Stage 2 writes to it incrementally.

```python
canvas["conversation_history"]   # list — appended after every answer
canvas["dimensions_covered"]     # dict — booleans, updated after cluster evaluation
canvas["idea"]                   # dict — draft-populated from extracted_signals
canvas["audience"]               # dict — draft-populated from extracted_signals
canvas["market"]                 # dict — draft-populated from extracted_signals
canvas["solution"]               # dict — draft-populated from extracted_signals
canvas["competition"]            # dict — draft-populated from extracted_signals
canvas["business_model"]         # dict — draft-populated from extracted_signals
canvas["founder"]                # dict — draft-populated from extracted_signals
canvas["risks"]                  # list — draft-populated from extracted_signals
```

Fields written during Stage 2 are treated as **drafts**. The synthesis call in Stage 3 overwrites them with authoritative values derived from the full conversation history. Stage 2 draft values are never shown to the founder directly.

**Layer 2 — Interview UI State** (`st.session_state`, separate from canvas)
These keys manage the interview loop. They do not persist beyond the session.

```python
st.session_state.current_cluster = {
    "dimension": str,           # which dimension this cluster covers
    "questions": [ ... ],       # full list of question dicts generated by LLM
    "answers": [ ... ],         # parallel list, filled as founder answers
    "current_index": int,       # which question is currently shown (0-based)
    "is_probe": bool            # True if this is a follow-up probe cluster
}
# None when Stage 2 first loads — triggers first cluster generation

st.session_state.interview_ready = bool
# False during LLM generation calls, True when cluster is ready to display
```

## 4. The Cluster Object

A cluster is the atomic unit of Stage 2. It represents one thematic group of questions targeting one dimension.

```python
cluster = {
    "dimension": "problem_discovery",
    "is_probe": False,
    "questions": [
        {
            "id": "q1",
            "question": str,          # personalized question text
            "format": str,            # "open_text" | "multiple_choice" | "scale" | "yes_no"
            "options": list | None,   # for multiple_choice and scale
            "placeholder": str | None,
            "helper_text": str | None,
            "yes_label": str | None,  # for yes_no only
            "no_label": str | None,   # for yes_no only
            "targets": str            # canvas field this answer maps to e.g. "idea.problem"
        },
        # ... 1-3 more questions
    ],
    "answers": [None, None, None],    # same length as questions, filled during PRESENTING
    "current_index": 0
}
```

**Cluster size rules** (enforced in prompt, not in code):
- Main dimension clusters: 2–4 questions
- Probe clusters: 1–2 questions
- Format decided by what the question is asking, not for variety

## 5. Cluster Lifecycle

Every cluster moves through four states. The UI and engine coordinate through `session_state`.

```
GENERATING
  One LLM call produces the full cluster dict.
  UI shows a loading/spinner state.
  On completion: cluster stored in session_state, st.rerun()
        ↓
PRESENTING
  UI reads current_index from cluster.
  Renders the question at that index using the correct render_* function.
  Founder submits answer → stored in cluster["answers"][current_index].
  current_index incremented.
  If more questions remain → st.rerun() (stay in PRESENTING).
  If all answered → move to EVALUATING.
        ↓
EVALUATING
  One LLM call evaluates all answers in the completed cluster.
  Returns: covered, confidence, probe_needed, probe_reason, extracted_signals.
  apply_extracted_signals() merges signals into canvas draft fields.
  dimensions_covered[dimension] updated.
  Answers appended to conversation_history.
  UI shows a loading/spinner state during this call.
        ↓
DONE
  Three possible next steps based on evaluation result:

  1. All 7 dimensions covered:
     canvas["stage"] = "synthesis" → st.rerun() → exits Stage 2

  2. Probe needed:
     generate_probe_cluster() called.
     Result set as session_state.current_cluster.
     → st.rerun() → back to PRESENTING

  3. Move to next dimension:
     generate_cluster() called for next uncovered dimension.
     Result set as session_state.current_cluster.
     → st.rerun() → back to GENERATING then PRESENTING
```

## 6. `interview_engine.py` — Functions

Four public functions. The module has no Streamlit imports.

---

### `generate_cluster(canvas) -> dict`

**When called:** At the start of the session, and after each completed non-probe evaluation that doesn't trigger a probe.

**What it does:**
- Builds `canvas_summary` from current canvas state (non-null fields only, readable format)
- Identifies the next dimension to cover — the first one with `dimensions_covered[dim] == False`
- Calls LLM with GENERATE_CLUSTER prompt
- Returns the full cluster dict

**LLM call settings:** temperature 0.7, expects JSON response matching cluster schema

**Important:** The function picks the dimension order. Suggested order: follow the natural logic of a startup conversation — problem → audience → market → solution → competition → business model → founder/risk. But if an earlier answer already covers a later dimension partially, the prompt should be allowed to adjust focus. The dimension order is a default, not a hard rule.

---

### `generate_probe_cluster(canvas, dimension, probe_reason) -> dict`

**When called:** After an evaluation returns `probe_needed: True`.

**What it does:**
- Builds a focused context string of only the exchanges for the dimension being probed
- Calls LLM with GENERATE_PROBE_CLUSTER prompt
- Returns a small cluster dict with `is_probe: True` and 1–2 questions

**LLM call settings:** temperature 0.7, expects JSON matching cluster schema

---

### `evaluate_cluster(canvas, cluster) -> dict`

**When called:** After all questions in a cluster are answered.

**What it does:**
- Packages cluster questions and answers as `qa_pairs`
- Builds `prior_signals` — what was already known about this dimension from earlier clusters
- Calls LLM with EVALUATE_CLUSTER prompt
- Returns evaluation result dict
- **Also calls `apply_extracted_signals(canvas, result["extracted_signals"])`** before returning — the canvas is updated inside this function as a side effect

**LLM call settings:** temperature 0.3, expects JSON response matching evaluation schema

**After this function returns**, the caller (`render_interview`) must also:
- Append all cluster Q&A pairs to `canvas["conversation_history"]`
- Update `canvas["dimensions_covered"][dimension]` based on `covered` field

---

### `is_complete(canvas) -> bool`

**When called:** After every cluster evaluation, before deciding next step.

**What it does:** Returns `all(canvas["dimensions_covered"].values())`

No LLM call. Pure Python.

## 7. `ui/interview.py` — Render Logic

One public function: `render_interview()`. No business logic inside — only session_state reads/writes and render_* calls.

---

### `render_interview()` — Control Flow

```
Entry point called by app.py when canvas["stage"] == "interview"

Step 1 — Initialize if needed:
  If "current_cluster" not in st.session_state:
    Set st.session_state.current_cluster = None
    Set st.session_state.interview_ready = False

Step 2 — Generate first cluster if none exists:
  If st.session_state.current_cluster is None:
    Show loading state
    cluster = generate_cluster(canvas)
    st.session_state.current_cluster = cluster
    st.session_state.interview_ready = True
    st.rerun()

Step 3 — Render current question:
  cluster = st.session_state.current_cluster
  idx = cluster["current_index"]
  question_dict = cluster["questions"][idx]
  
  Call the correct render_* function based on question_dict["format"]
  Pass: question, options, placeholder, helper_text, yes_label, no_label
  Pass: question_number = (total answered so far + 1)

Step 4 — Handle submission:
  When founder submits:
    Store answer in cluster["answers"][idx]
    cluster["current_index"] += 1
    
    If cluster["current_index"] < len(cluster["questions"]):
      st.rerun()  # show next question in same cluster
      return
    
    # All questions answered — evaluate
    Show loading state ("Thinking about your answers...")
    result = evaluate_cluster(canvas, cluster)
    # canvas draft fields already updated inside evaluate_cluster
    
    # Append to conversation_history
    for qa in build_qa_pairs(cluster):
      canvas["conversation_history"].append({
        "dimension": cluster["dimension"],
        "question": qa["question"],
        "question_format": qa["format"],
        "answer": qa["answer"],
        "probe_triggered": cluster["is_probe"]
      })
    
    # Update coverage
    canvas["dimensions_covered"][result["dimension"]] = result["covered"]
    
    # Decide next step
    If is_complete(canvas):
      canvas["stage"] = "synthesis"
      st.rerun()
      return
    
    If result["probe_needed"]:
      Show loading state
      probe = generate_probe_cluster(canvas, result["dimension"], result["probe_reason"])
      st.session_state.current_cluster = probe
      st.rerun()
      return
    
    # Move to next dimension
    Show loading state ("Moving to the next topic...")
    next_cluster = generate_cluster(canvas)
    st.session_state.current_cluster = next_cluster
    st.rerun()
```

### Question Number Tracking

The founder sees a question number (e.g. "Question 7") but not which dimension they're on. Track total questions answered globally across all clusters:

```python
# Compute on each render — don't store separately
total_answered = sum(
    len(c["answers"])  # only count completed clusters
    for c in st.session_state.get("completed_clusters", [])
) + cluster["current_index"]
```

Store completed clusters in `st.session_state.completed_clusters = []`. Append each cluster to this list after evaluation before generating the next one.

## 8. The Three Prompts

All three live in `prompts.py` as string constants with `{placeholder}` slots filled at call time.

---

### `INTERVIEW_SYSTEM`

Used as the system message in all three LLM calls. Static — filled once at call time with `raw_input` and `canvas_summary`.

```
You are a startup mentor conducting a validation interview with an 
early-stage founder.

Your role:
- Understand their idea deeply through sharp, specific questions
- Challenge assumptions without being hostile
- Seek real evidence, not opinions or hypotheticals
- Sound like a knowledgeable human mentor — not a form or a chatbot

Rules you never break:
- Never mention Lean Canvas, Mom Test, YC, or any framework by name
- Never refer to "dimensions", "clusters", or "coverage"
- Never ask for information the founder has already given
- Never cheerlead or validate — stay neutral and curious
- Personalize every question to this specific idea and founder
- Do not copy questions verbatim from any template — adapt them

The founder's idea:
{raw_input}

What you already know about them:
{canvas_summary}
```

---

### `GENERATE_CLUSTER_PROMPT`

User message for `generate_cluster()` and `generate_probe_cluster()` (probe uses a modified version).

```
Your task is to generate the next cluster of interview questions.

Dimension to cover: {dimension}
This is a probe cluster: {is_probe}

Coverage status across all dimensions:
{dimensions_covered}

Conversation so far:
{conversation_history_summary}

Reference framework — use this to understand what needs to be learned 
for each dimension. Do NOT copy these questions verbatim. Use them as 
a thinking guide. Personalize every question to this founder's idea:
{question_bank}

Generation rules:
- Generate 2-4 questions for a main cluster, 1-2 for a probe cluster
- Pick the format that fits what you are asking — do not choose format 
  for variety
- For yes_no questions: write yes_label and no_label that are specific 
  to this question (e.g. "Seen it firsthand" not just "Yes")
- Add helper_text only when the question risks a vague or buzzword answer
- Add placeholder only when a concrete example would clarify the 
  expected answer format — make it feel real and specific to their idea
- Omit helper_text and placeholder when the question is clear without them
- If earlier answers already partially cover this dimension, focus only 
  on the gaps
- Questions must feel like a natural continuation of the conversation

{probe_context}
(Only present for probe clusters:
"Probe reason: {probe_reason}
Questions and answers already given for this dimension:
{dimension_exchanges}")

Return JSON only — no explanation, no markdown:
{
  "dimension": str,
  "is_probe": bool,
  "questions": [
    {
      "id": str,
      "question": str,
      "format": "open_text" | "multiple_choice" | "scale" | "yes_no",
      "options": list | null,
      "placeholder": str | null,
      "helper_text": str | null,
      "yes_label": str | null,
      "no_label": str | null,
      "targets": str
    }
  ]
}
```

## 10. Error Handling

Two failure points in Stage 2:

**LLM returns invalid JSON:**
Wrap every LLM call in a retry loop — attempt JSON parse, if it fails, retry once with an appended instruction: "Your previous response was not valid JSON. Return JSON only, no explanation, no markdown fences." If it fails twice, surface a user-visible error in the UI: "Something went wrong generating the next question. Please refresh and try again."

**LLM returns a cluster with 0 questions:**
Validate the returned cluster before storing in session_state. If `len(cluster["questions"]) == 0`, treat as a generation failure and retry once.

Both retries should be handled inside `interview_engine.py` — the UI should never see a raw exception from an LLM call.

## 11. Build Order

Build and test each step before moving to the next. Each step is independently testable.

**Step 1 — Question bank and prompts**
Write `QUESTION_BANK`, `INTERVIEW_SYSTEM`, `GENERATE_CLUSTER_PROMPT`, `GENERATE_PROBE_CLUSTER_PROMPT`, and `EVALUATE_CLUSTER_PROMPT` in `prompts.py`. No code yet. Test the prompts manually in a playground with sample canvas states before wiring them up.

**Step 2 — Canvas helpers**
Write `build_canvas_summary()`, `build_qa_pairs()`, `apply_extracted_signals()`, `get_next_dimension()` in `interview_engine.py`. Write unit tests for `apply_extracted_signals()` — it's the most error-prone helper. Test with null values, partial values, nested dicts, and the `risks[]` list case.

**Step 3 — `generate_cluster()`**
Wire up the first LLM call. Test it against 3 different raw_input ideas. Verify the returned JSON matches the cluster schema. Verify personalization — questions should reference the specific idea, not be verbatim from the bank.

**Step 4 — `evaluate_cluster()`**
Wire up the evaluation call. Test it with 5 answer quality scenarios: perfect answers, vague answers, hypothetical answers, contradictory answers, minimal one-word answers. Verify `extracted_signals` populates correctly and `apply_extracted_signals()` writes to canvas correctly.

**Step 5 — `generate_probe_cluster()`**
Wire up the probe call. Test by feeding it a `probe_reason` from a real evaluation result. Verify probe questions feel like natural follow-ups, not repetitions.

**Step 6 — `is_complete()`**
One line. Test it.

**Step 7 — `render_interview()` skeleton**
Build the UI control flow without LLM calls — hardcode a sample cluster in session_state and verify the question-by-question rendering works, answers store correctly, and index advances correctly.

**Step 8 — Full integration**
Connect the engine to the UI. Run a full mock interview from start to finish. Manually submit ~15 answers across all dimensions. Verify: all 7 dimensions covered, conversation_history populated, canvas draft fields written, stage transitions to "synthesis".

**Step 9 — Edge cases**
Test: very short answers ("yes", "idk"), answers that cover multiple dimensions at once, contradictory answers across clusters, all dimensions covered by cluster 4 (early exit), probe chains (probe of a probe).
